# Revert Domain Emitter — Phase 1 Teardown Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove the Phase-1 domain emitter artifacts (`domain.ts`, `conformance.ts`, `view-overlay.ts`) from visual-editor; replace the 4 live domain-function call sites with temporary generic bridges (`ensureMemberArray`/`getMemberArray`) while the proper typed replacement (`domain-ops.ts` over `Dehydrated<T>`) is built as a follow-on.

**Architecture:** The store substrate has always been `AstNodeModel<T>` (a recursive mapped type in `types.ts` that strips Langium internals). Phase 1 generated a parallel 5201-line `@ts-nocheck` file with 593 `any`-typed functions — concept right, output wrong. The 4 live functions are one-liners that duplicate already-existing helpers in `node-projection.ts`. Teardown is pure deletion + 4 inline replacements; no new code needed in this plan.

**What replaces `domain.ts` (follow-on plan):** `domain-ops.ts` — properly typed mutators and normalizing readers over `Dehydrated<T>`, generated by the langium-zod emitter. See §Addendum for the full follow-on architecture.

**Tech Stack:** TypeScript, Vitest, pnpm workspace. No cross-repo changes required (langium-zod retains the domain emitter capability; rune just stops using the bad artifact).

## Addendum: Follow-on Architecture (domain-ops.ts + Dehydrated<T> + RuneStoreHydrator)

These are **follow-on tasks** — the 5 teardown tasks above are self-contained and must complete first. This addendum documents the confirmed replacement architecture so a follow-on agent has full context.

---

### A. `Dehydrated<T>` — the correct store substrate type

`AstNodeModel<T>` in `types.ts` has a gap: it strips Langium internals but passes `Reference<T>` through untouched (keeps the full live reference object). `Dehydrated<T>` fixes this:

```ts
// packages/visual-editor/src/types.ts — add alongside AstNodeModel<T>
type LangiumRuntime = '$container' | '$containerProperty' | '$containerIndex' | '$cstNode' | '$document';

export type Dehydrated<T extends AstNode> = {
  [K in keyof Omit<T, LangiumRuntime>]:
    NonNullable<T[K]> extends AstNode        ? Dehydrated<NonNullable<T[K]>> :
    NonNullable<T[K]> extends AstNode[]      ? Dehydrated<NonNullable<T[K]>[number]>[] :
    NonNullable<T[K]> extends Reference<any> ? { $refText: string } :
    T[K]
};
```

`AstNodeModel<T>` can be retired once all consumers are switched to `Dehydrated<T>` — that is a separate Phase-3 cutover task, not part of this teardown.

---

### B. `RuneStoreHydrator` — parse boundary for the store

`DefaultHydrator` as-is does **not** produce a clean store wire model:
1. **CST is retained** — `dehydrateAstNode` calls `this.dehydrateCstNode(node.$cstNode, context)` and writes the full CST (grammar element IDs, token types, byte offsets) into `obj.$cstNode`.
2. **Cross-refs include `$refNode`** — `dehydrateReference` produces `{ $refText, $refNode: ephemeralIndex }` where `$refNode` is an index into a per-call BiMap (meaningless outside that call).
3. **`hydrateReference` requires `$refNode`** — uses a non-null assertion `context.cstNodes.get(reference.$refNode)!`; omitting `$refNode` → runtime failure.

**Three overrides fix all three:**

```ts
// packages/visual-editor/src/store/rune-store-hydrator.ts
// SPDX-License-Identifier: MIT
// Copyright (c) 2026 Pradeep Mouli

import type { AstNode, CstNode, Reference } from 'langium';
import type { DehydrateContext, HydrateContext } from 'langium/serializer';
import { DefaultHydrator } from 'langium/serializer';

export class RuneStoreHydrator extends DefaultHydrator {
  /**
   * Return an empty object — no recursion, no BiMap registration, no CST data.
   * Overriding HERE (not dehydrateAstNode + delete) avoids the expensive traversal
   * that super.dehydrateAstNode would otherwise run before we could clean up.
   */
  protected override dehydrateCstNode(_node: CstNode, _context: DehydrateContext): unknown {
    return {};
  }

  /** Strip the ephemeral $refNode index — only $refText survives into the wire model. */
  protected override dehydrateReference(reference: Reference, _context: DehydrateContext): unknown {
    return { $refText: reference.$refText };
  }

  /**
   * Dehydrate-only in practice (store nodes are mutated directly, never re-hydrated
   * from JSON). Override for correctness: use $refText only; linker resolves lazily.
   */
  protected override hydrateReference(
    reference: { $refText: string },
    node: AstNode,
    name: string,
    _context: HydrateContext
  ): Reference {
    return this.linker.buildReference(node, name, undefined!, reference.$refText);
  }
}
```

**No source location in the wire model.** Source location (byte offsets, line/column) is NOT stored. For "jump to source" / location-sensitive editor features, use `services.workspace.AstNodeLocator` on the LSP server via a live round-trip — it takes the dehydrated node and returns the live source location from the current parse result. This avoids stale offsets when the document is edited.

**Register via `RuneDslModule` DI override:**
```ts
// In the DI module that assembles the visual-editor services:
hydrator: () => new RuneStoreHydrator(/* inject linker */),
```

### C. `domain-ops.ts` — typed replacement for `domain.ts`

`domain.ts` had the right concept (per-type domain operations) but wrong execution (`@ts-nocheck`, 593 `any`-typed functions). The replacement is `domain-ops.ts`, generated by `langium-zod` over `Dehydrated<T>`.

**Surface: three function families per type:**

| Family | Naming | Returns | Side effects |
|--------|--------|---------|--------------|
| `get*` | `getAttributes(node)` | `Dehydrated<X>[]` — always returns `[]`, never `undefined` | none |
| `add*` | `addAttribute(node, attr)` | `void` | pushes to member array, initializes `[]` if absent |
| `remove*At` | `removeAttributeAt(node, idx)` | `void` | `node.attributes?.splice(idx, 1)` |

**Example signatures over `Dehydrated<T>`:**
```ts
// Data / Annotation / Choice → .attributes
export function getAttributes(node: Dehydrated<Data>): Dehydrated<Attribute>[] {
  return node.attributes ?? [];
}
export function addAttribute(node: Dehydrated<Data>, attr: Dehydrated<Attribute>): void {
  (node.attributes ??= []).push(attr);
}
export function removeAttributeAt(node: Dehydrated<Data>, idx: number): void {
  node.attributes?.splice(idx, 1);
}

// RosettaEnumeration → .enumValues
export function getEnumValues(node: Dehydrated<RosettaEnumeration>): Dehydrated<RosettaEnumerationValue>[] {
  return node.enumValues ?? [];
}
export function addEnumValue(node: Dehydrated<RosettaEnumeration>, val: Dehydrated<RosettaEnumerationValue>): void {
  (node.enumValues ??= []).push(val);
}
export function removeEnumValueAt(node: Dehydrated<RosettaEnumeration>, idx: number): void {
  node.enumValues?.splice(idx, 1);
}

// RosettaFunction → .inputs
export function getFunctionInputs(node: Dehydrated<RosettaFunction>): Dehydrated<Attribute>[] {
  return node.inputs ?? [];
}
export function addFunctionInput(node: Dehydrated<RosettaFunction>, input: Dehydrated<Attribute>): void {
  (node.inputs ??= []).push(input);
}
export function removeFunctionInputAt(node: Dehydrated<RosettaFunction>, idx: number): void {
  node.inputs?.splice(idx, 1);
}

// RosettaRecordType → .features
export function getRecordFeatures(node: Dehydrated<RosettaRecordType>): Dehydrated<Attribute>[] {
  return node.features ?? [];
}
// ... addRecordFeature, removeRecordFeatureAt analogously
```

These functions are plain typed functions — no class instances — so they are compatible with Zustand/Zundo/Mutative drafts.

### D. `node-projection.ts` ownership after teardown

After Tasks 1–5, `node-projection.ts` owns ONLY:
- Node-id construction (`makeNodeId`, `nameFromNodeId`, `splitNodeId`)
- Edge-id construction/parsing (`makeEdgeId`, `parseEdgeId`)
- `GraphMetadata` field-set and `stripGraphMetadata` / `withGraphMetadata` / `astRelevantProjection`
- Array↔Map derivation primitives (`toNodesById`, `nodesFromMap`, `toEdgesById`, `edgesFromMap`)
- Re-export of `resolveNodeKind` from `model-helpers.ts`

`getMemberArray` / `ensureMemberArray` (currently temporary bridge) migrate to `rune-store-hydrator.ts` as the co-located `MEMBER_FIELD_BY_KIND` map, once `domain-ops.ts` exists. At that point the editor-store call sites switch from generic helpers to typed `domain-ops.ts` functions, and the generic helpers can be removed from `node-projection.ts`.

### E. `DefaultJsonSerializer` — not needed for store

`DefaultJsonSerializer` throws on plain objects not attached to a Langium document (`getDocument()` raises). The existing `model-to-ast.ts` → `serializeRuneModel` path handles final `.rosetta` text emission. No new serialization layer is needed.
